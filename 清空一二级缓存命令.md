# 清空一二级缓存

## ⚠️ 重要提示

**不要使用 `FLUSHALL` 命令！** 这会删除 Redis 中的所有数据，包括 `documents:processing` Stream 和消费者组，导致 Python 文档处理消费者持续报错：
```
NOGROUP No such key 'documents:processing' or consumer group 'python-doc-processor'
```

---

## 推荐方式：通过前端按钮清空缓存

前端 Chat 页面右上角有两个缓存清空按钮：
- 🗑️ **橙色按钮** — 清空一级缓存（Redis），只删除 `cache:*` 前缀的 key
- 🗑️ **紫色按钮** — 清空二级缓存（pgvector 语义缓存）

这两个按钮调用的是后端的安全接口，不会误删系统数据。

---

## 命令行方式（仅当无法使用前端时）

### 1. 清空 Redis 一级缓存（安全方式）

```bash
# ✅ 正确：只删除缓存 key，不影响 Stream 等系统 key
docker exec -it rag-redis redis-cli EVAL "return redis.call('del', unpack(redis.call('keys', 'cache:*')))" 0

# 或者使用 SCAN 模式（生产环境推荐，不会阻塞 Redis）
docker exec -it rag-redis redis-cli EVAL "
  local cursor = '0'
  local count = 0
  repeat
    local result = redis.call('SCAN', cursor, 'MATCH', 'cache:*', 'COUNT', '100')
    cursor = result[1]
    local keys = result[2]
    if #keys > 0 then
      count = count + #keys
      redis.call('del', unpack(keys))
    end
  until cursor == '0'
  return count
" 0
```

### 2. 清空 pgvector 二级缓存

```bash
docker exec -it rag-postgres psql -U postgres -d rag_kb -c "DELETE FROM cache_entries;"
```

---

## 如果不小心执行了 FLUSHALL

如果已经执行了 `FLUSHALL` 导致 `NOGROUP` 错误，**无需重启服务**。Python consumer 已经具备自动恢复能力：

1. Consumer 检测到 `NOGROUP` 错误后会自动调用 `xgroup_create` 重建 Stream 和消费者组
2. 重建完成后，新的文档处理任务可以正常入队和处理
3. 之前未处理完的消息会丢失（FLUSHALL 删除了所有消息），需要重新上传文档

等待约 5-10 秒后，错误日志会自动消失。
